# architecture-notes

アーキテクチャを理解し、比較し、実務で選択できるようにするためのノート集です。

このリポジトリでは、有名なソフトウェアアーキテクチャ、設計パターン、システム構成を以下の観点で整理します。

- 何を解決するための考え方か
- どのような構成要素で成り立つか
- どのような場面で向いているか、向いていないか
- ほかのアーキテクチャと何が違うか
- Mermaid図で見るとどう表現できるか
- 参考にした文献、公式ドキュメント、論文、記事

## ドキュメント方針

各アーキテクチャの説明は、原則として [docs/_template.md](./docs/_template.md) に沿って作成します。

特に重視することは次のとおりです。

- 初学者にもわかる短い概要を書く
- 実務で迷いやすい判断軸を書く
- Mermaidで構造やデータの流れを図示する
- 類似アーキテクチャとの差分を書く
- 参照した資料を必ず「参考」に明記する

## 読み方

最初に全体像をつかみたい場合は、[アーキテクチャ横断比較](./docs/横断比較.md) から読むと、目的別に候補を絞りやすくなります。用語の意味を確認したい場合は、[用語集](./docs/用語集.md) を参照してください。

各カテゴリの作成済みドキュメント一覧は、カテゴリ配下の `README.md` を正とします。全体から探したい場合は、[アーキテクチャ一覧](./docs/一覧.md) を入口にしてください。

文書を追加・更新するときの到達ラインは、[ドキュメント品質基準](./docs/品質基準.md) にまとめています。

## 追加時の流れ

1. 配置先カテゴリを決める
2. 対応する `docs/<カテゴリ名>/` にMarkdownファイルを作成する
3. カテゴリREADMEの作成済みドキュメント一覧に追加する
4. [docs/_template.md](./docs/_template.md) に沿って概要、図、比較、参考を書く
5. `npm run docs:list` で [docs/一覧.md](./docs/一覧.md) を更新する
6. 品質チェックを実行し、[ドキュメント品質基準](./docs/品質基準.md) の完了前チェックを満たしているか確認する

## 品質確認

更新時は、少なくとも次の観点を確認します。

- カテゴリREADMEから個別文書へリンクできる
- 個別アーキテクチャ文書に `## 参考` と Mermaid図がある
- 類似アーキテクチャとの差分が具体的で、汎用表現に逃げていない
- 図の説明が対象固有の読み方になっている
- 表記ゆれが用語集やカテゴリREADMEと大きくずれていない

自動チェックは `npm run check` で実行できます。GitHub Actionsでも同じ観点を確認します。
