# Zero Trust Architecture

## 概要

ネットワーク内外を問わず暗黙に信頼せず、利用者、端末、アプリケーション、通信を継続的に検証する設計です。

## 解決したい課題

- 侵害や障害を完全には避けられない前提で、影響範囲を抑える設計が必要です。
- 変更影響、運用負荷、理解しやすさのバランスを取る
- 適用範囲と責務境界を明確にする

## 基本構成

| 要素 | 責務 |
| --- | --- |
| Identity | ユーザー、サービス、端末の識別情報 |
| Policy Engine | アクセス可否を判断する仕組み |
| Policy Enforcement | 判断結果を通信や操作に適用する仕組み |
| Continuous Monitoring | 状態変化を監視し継続評価する仕組み |

## 向いている場面

- 重要資産の保護、障害隔離、連鎖障害の抑制を重視する場面に向きます。
- 変更や障害の影響範囲を意識して設計したい
- チーム内で構成要素の責務を共通認識にしたい

## 向いていない場面

- 課題が小さく、導入コストのほうが大きい
- 境界や責務を運用で守る体制がない
- 名前だけ導入して実装方針やレビュー観点が変わらない

## メリット

- 責務の分離により変更箇所を見つけやすい
- 設計判断の観点をチームで共有しやすい
- 適用条件が合えば、保守性や拡張性を高めやすい

## デメリット

- 抽象化や構成要素が増え、初期コストがかかる
- 境界設計を誤ると、かえって複雑になる
- 小さなシステムでは過剰設計になりやすい

## 実務での判断ポイント

- 何を守りたいのか、何を変えやすくしたいのかを先に決める
- 導入後に責務境界をレビューできるルールを用意する
- 既存システムへは小さな範囲から適用し、効果を確認する
