# データ・更新モデル系

データの保存、更新、参照、処理パイプラインを扱うカテゴリです。

## 比較の早見表

| 判断したいこと | 候補 | 見るポイント |
| --- | --- | --- |
| 更新と参照のモデルを分けたい | CQRS | Read Modelの同期遅延と再構築手順を設計する |
| 状態変更の履歴を正本として残したい | Event Sourcing | イベントスキーマ、再生、Snapshot、個人情報削除を考える |
| 複数サービスの更新を整合させたい | Sagaパターン | 補償処理、冪等性、途中状態、タイムアウトを設計する |
| 分析やBIへデータを届けたい | Data Pipeline Architecture | データ所有者、品質、再処理、データ系譜を管理する |
| バッチとリアルタイムを併用したい | Lambda Architecture | 2系統の処理ロジックを保守できるか確認する |
| ストリーム処理に統一したい | Kappa Architecture | ログ保持、再生コスト、遅延イベントの扱いを確認する |
| データ所有権をドメインへ分散したい | Data Mesh | 組織、ガバナンス、セルフサービス基盤が必要 |
| レイクとDWHを統合したい | Lakehouse Architecture | テーブル形式、カタログ、権限、性能要件を確認する |

## 作成済みドキュメント

| アーキテクチャ | 概要 |
| --- | --- |
| [CQRS](./CQRS.md) | 更新を扱うCommandと参照を扱うQueryの責務を分ける設計です。 |
| [Event Sourcing](./イベントソーシング.md) | 現在状態ではなく、状態を変化させたイベント列を保存し、そこから状態を復元する設計です。 |
| [Sagaパターン](./Sagaパターン.md) | 分散トランザクションを複数のローカルトランザクションと補償処理で扱う設計です。 |
| [Data Pipeline Architecture](./データパイプラインアーキテクチャ.md) | データの収集、変換、検証、保存、配信を段階的な処理として設計する構成です。 |
| [Lambda Architecture](./ラムダアーキテクチャ.md) | 正確性を重視するバッチ層と低遅延を重視するスピード層を併用する構成です。 |
| [Kappa Architecture](./カッパアーキテクチャ.md) | バッチ層を分けず、ストリーム処理を中心にデータ処理を統一する構成です。 |
| [Data Mesh](./データメッシュ.md) | ドメインごとにデータをプロダクトとして所有し提供する、組織と技術をまたぐ考え方です。 |
| [Lakehouse Architecture](./レイクハウスアーキテクチャ.md) | データレイクの柔軟性とデータウェアハウスの管理性を統合しようとするデータ基盤構成です。 |
