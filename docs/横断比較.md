# アーキテクチャ横断比較

## 概要

この文書は、個別のアーキテクチャを読む前に「どの観点で選ぶか」を整理するための横断比較です。アーキテクチャ名だけで選ぶのではなく、解決したい課題、変更頻度、チーム体制、運用能力、データ整合性の要求から候補を絞ることを目的にします。

## まず見る判断軸

| 判断軸 | 確認すること | 関連しやすいカテゴリ |
| --- | --- | --- |
| 責務分離 | 業務ルール、UI、DB、外部APIをどこで分けるか | レイヤー・責務分離系 |
| 分割単位 | 1つのアプリで保つか、サービスやモジュールへ分けるか | 分散システム系 |
| データ整合性 | 強整合、最終的整合性、履歴、監査をどう扱うか | データ・更新モデル系 |
| UIの複雑さ | 画面状態、部品、フロントエンド境界をどう管理するか | フロントエンド系 |
| システム連携 | API、キュー、イベント、移行をどう接続するか | エンタープライズ統合・連携系 |
| 運用・リリース | デプロイ、スケール、通信基盤、観測性をどう設計するか | インフラ・運用系 |
| 障害・侵害 | 障害影響、連鎖障害、セキュリティ境界をどう抑えるか | セキュリティ・信頼性系 |

## よくある選択の分岐

| 状況 | まず検討する候補 | 注意点 |
| --- | --- | --- |
| 小さく始めたい | モノリシックアーキテクチャ、Three-Tier Architecture、レイヤードアーキテクチャ | 最初から分散しすぎない。内部境界だけは早めに意識する |
| モノリスが大きくなってきた | モジュラーモノリス、Modular Architecture、Vertical Slice Architecture | いきなりマイクロサービス化せず、変更理由と所有境界を先に見つける |
| 業務ルールが複雑 | DDDアーキテクチャ、クリーンアーキテクチャ、オニオンアーキテクチャ | DDDの用語だけを導入しても効果は薄い。ドメイン専門家との会話が必要 |
| 複数サービスへ分割したい | マイクロサービスアーキテクチャ、API Gateway Pattern、Sagaパターン | 監視、CI/CD、契約管理、データ所有権が先に必要 |
| 非同期処理を増やしたい | イベント駆動アーキテクチャ、Message Queue Architecture、Publish-Subscribe | 重複、順序、再試行、冪等性を設計する |
| 読み取り性能を上げたい | CQRS、Data Pipeline Architecture、Lakehouse Architecture | 更新モデルと参照モデルの同期遅延を説明できるようにする |
| UIが複雑になってきた | MVVM、Flux、Redux Architecture、Feature-Sliced Design | グローバル状態を増やしすぎない。局所状態との境界を決める |
| レガシーを段階移行したい | Strangler Fig Pattern、API Gateway Pattern、Backend for Frontend | 新旧データ同期と移行完了条件を先に決める |
| リリースリスクを下げたい | Blue-Green Deployment、Canary Release、Feature Flag | DB変更の後方互換性とロールバック条件を設計する |
| 障害影響を局所化したい | Bulkhead Pattern、Circuit Breaker Pattern、Cell-Based Architecture | 何を守るか、どの単位で隔離するかを明確にする |

## 過剰設計になりやすい組み合わせ

| 組み合わせ | 起きやすい問題 | 見直しポイント |
| --- | --- | --- |
| 小規模CRUD + クリーンアーキテクチャ全面適用 | DTO、Port、UseCaseが多く、変更量だけ増える | 業務ルールが厚い場所だけ境界を強める |
| 初期プロダクト + マイクロサービス | 開発より運用、監視、契約管理の負担が大きくなる | まずモジュラーモノリスで境界を検証する |
| 単純な参照画面 + CQRS | Read Model同期や再構築の手間が価値を上回る | 参照性能やモデル差が本当に大きいか確認する |
| 小規模UI + Redux全面適用 | Action、Reducer、Storeの管理が重くなる | 共有状態だけをStoreへ置き、局所状態はUI側に残す |
| 少数サービス + Service Mesh | Proxy、証明書、制御プレーンの運用が重い | API Gatewayやライブラリで足りないか確認する |

## 実務での使い方

- まず「何が痛いのか」を1文で書く
- その痛みが責務、分散、データ、UI、連携、運用、信頼性のどれに近いかを分類する
- 候補を2つから3つに絞り、向いていない場面とデメリットを先に読む
- 導入後に守るルールを、レビュー観点やディレクトリ構成に落とす
- 途中で合わないとわかったら、より単純な構成へ戻す余地を残す
