# ドキュメント品質基準

## 概要

このリポジトリでは、若手が概要をつかめて、実務経験者が判断材料として使える水準を目指します。単に用語を説明するだけではなく、適用条件、トレードオフ、失敗しやすい点、類似アーキテクチャとの差分まで書きます。

## 到達ライン

| 観点 | 必須ライン |
| --- | --- |
| 概要 | 初学者が「何のための考え方か」を説明できる |
| 課題 | 放置すると何が困るのかが具体的に書かれている |
| 構成 | 要素名だけでなく、実務上の責務が書かれている |
| 図 | Mermaidで全体像または流れを表している |
| 向き不向き | 適用すべき場面と避けるべき場面が条件付きで書かれている |
| 適用例 | 実務で想像しやすい短い適用例が書かれている |
| 比較 | 似た用語との違いが1文以上で具体化されている |
| 実務判断 | 導入前に確認する条件、運用、撤退条件が書かれている |
| 参考 | 公式ドキュメント、原典、論文、書籍、信頼できる技術記事が明記されている |

## 表記・命名

- 文書タイトルは、広く英語名で流通しているものは英語名を使い、本文中で日本語の意味を補足する。
- ファイル名は日本語を基本にし、略称や固有名として定着しているものは英字を含めてもよい。
- 同じ文書内では、`API Gateway` と `APIゲートウェイ` のような表記を混在させない。初出で補足し、以降は主表記に揃える。
- READMEの一覧名、個別文書の見出し、用語集の表記がずれないようにする。
- 英語名を残す場合でも、実務判断の本文は日本語で説明する。

## 避ける表現

- 「関連する問題領域で使われる」のような差分になっていない比較
- 「責務の分離により変更箇所を見つけやすい」だけで終わる汎用メリット
- 「課題が小さく、導入コストのほうが大きい」だけで終わる汎用的な不向き条件
- 「主要コンポーネント」のように、図が対象アーキテクチャを説明していない状態
- 「名前だけ導入して実装方針が変わらない」のような、どの文書にも当てはまる文だけの注意
- 図の説明が「中心になる責務と流れを簡略化したものです」だけで終わり、対象固有の読み方を説明していない状態

## 推奨する書き方

- 初学者向けには、まず比喩よりも具体例を書く
- 実務者向けには、スケール、運用、組織、データ整合性、セキュリティの観点を書く
- 「常に」「必ず」は避け、条件と例外を書く
- 似た概念は表で比較する
- 失敗しやすい点は、原因と対策をセットで書く
- よくある適用例は、ECサイト、管理画面、社内基幹システム連携、分析基盤など、読者が導入範囲を想像できる粒度で書く
- 参考資料は、可能なら「原典・一次資料」「補助資料」「関連パターン」に分け、公式情報と解説記事を混ぜない

## レビュー観点

- 読者はこの文書だけで、導入すべきか保留すべきか判断できるか
- 図は文章の言い換えではなく、理解を助けているか
- 類似アーキテクチャとの差分は具体的か
- 参考資料は信頼できるか
- 過剰設計への警告が書かれているか
- 若手が読んでも怖くなく、ベテランが読んでも薄くないか

## 完了前チェック

- READMEとカテゴリREADMEから個別文書へリンクできる
- 個別アーキテクチャ文書に `## 参考` がある
- 個別アーキテクチャ文書に ` ```mermaid ` の図がある
- 新規文書には `## よくある適用例` がある
- 類似アーキテクチャとの差分が、対象ごとの違いとして書かれている
- 「何に向いていないか」が導入コストだけでなく、失敗条件として書かれている
- 図の直後に、その図をどう読むかが対象固有に説明されている
- 参考資料が本文の主張を支えるものになっており、一次資料と補助資料の区別がつく
